# お問い合わせフォーム

## プロジェクト概要
Node.js + Express + EJSで作るお問い合わせフォーム。データはSQLiteに保存する。

## 技術スタック
- 言語/フレームワーク: Node.js + Express + EJS
- データベース: SQLite
- テストランナー: Jest
- HTTPレベルの結合テスト: supertest（Expressの`app`を直接テストし、サーバー起動不要）

## 開発方針
- テスト駆動開発（TDD）を前提とする。「テストを書く→実装する→通す」のサイクルで進める。
- テストしやすくするため、役割を分離した構成にする。
  - `routes/`: リクエストの受け口（supertestで結合テスト）
  - `services/` や `validators/`: バリデーションやステータス遷移ルールなどの純粋関数（単体テスト）
  - `repositories/`: SQLiteへのDBアクセスをまとめる層
- テストはSQLiteを`:memory:`もしくはテストごとの一時ファイルにして、クリーンな状態から実行する。

## コーディング規約
- 言語仕様: `const`/`let`を使用し`var`は使わない。等価比較は`===`/`!==`を使う。非同期処理は`async/await`で統一する。
- モジュール形式: CommonJS（`require` / `module.exports`）を使用する。
- 命名規則: 変数・関数はcamelCase、クラスはPascalCase、定数はUPPER_SNAKE_CASE、ファイル名はkebab-case。
- フォーマッタ/リンター: Prettier（自動整形）+ ESLint（静的解析）を導入する。
- コメント: 「何をしているか」ではなく、非自明な「なぜそうしているか」がある場合のみ書く。
- エラーハンドリング: Expressのエラーハンドリングミドルウェアに集約する。
- セキュリティ: SQLは必ずプレースホルダ（パラメータ化クエリ）を使い、文字列結合しない。EJSでの出力は`<%= %>`（自動エスケープ）を基本とし、`<%- %>`（非エスケープ）は必要な場合のみ使う。
- テストファイル命名: `*.test.js`とし、対応する実装ファイルと名前を揃える。

## Git運用
- コミットメッセージはConventional Commits形式（`feat:`, `fix:`, `test:`, `refactor:`, `docs:`など）に従う。

## 機能
エンドユーザーが入力する「お問い合わせフォーム」と、管理者が確認する「管理ページ」からなる。

### お問い合わせフォーム
- 「名前」「メールアドレス」「件名」「本文」を入力して送信する。
- 送信前の「確認画面」有り。
- 送信後には「お問い合わせありがとうございました」を出力する。
- メール通知は行わない（画面表示のみで完結。SMTP等の外部メール送信は導入しない）。

### 管理ページ
- お問い合わせの一覧表示をする。
- 一覧でクリックしたら詳細が表示される。
- ステータスの管理機能があり、「新規」「対応中」「解決済み」の3段階をドロップダウンリストで変更できる。
- 認証（ログイン）は今回のスコープ外（未実装）。ただし将来追加しやすいよう、管理ページのルートは`/admin`配下にまとめて実装し、認証ミドルウェアを後から一括で差し込める構成にしておく。
