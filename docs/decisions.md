# 設計判断の決定ログ

CLAUDE.mdや各仕様ファイルに書いた「何を作るか」に対して、「なぜその設計・技術を選んだか」を時系列でまとめる。過去の対話で決めた内容を遡って記録している。

## 2026-07-10

### 技術スタック: Node.js + Express + EJS
新しい言語を覚える必要がなく、学習コストが最も低いという理由でNode.js + Express + EJSを採用した。Ruby on Rails・Laravel・Djangoも比較検討し、特にDjango Adminによる管理画面の自動生成という利点も検討したが、「知っている言語のまま進められること」を優先した。

### データベース: SQLite + better-sqlite3
個人開発規模でセットアップが簡単なことに加え、TDD前提の開発方針と相性が良い（`:memory:`でテストごとに高速・独立した状態を作れる）ため採用。SQLiteドライバは`node:sqlite`（Node組み込み、ビルド不要）も候補にあったが、より定番で情報量の多い`better-sqlite3`を選んだ。これにより、devcontainerにpython3等のネイティブビルド用ツールを追加する必要が生じたが、許容した。

### 管理ページの認証: 今回はスコープ外
管理ページの認証（ログイン）は今回のスコープ外とした。ただし、後から認証を追加する際の変更コストを抑えるため、管理ページのルートはすべて`/admin`配下にまとめる設計方針を採用した。こうしておけば、将来的に認証ミドルウェアを1箇所（`router.use(requireAuth)`相当）追加するだけで済み、既存の一覧/詳細/ステータス変更ロジックは無修正で対応できる見込み。

### メール通知: 実装しない
問い合わせ送信後は画面表示（「お問い合わせありがとうございました」）のみで完結させる方針とした。外部メール送信の仕組み（SMTP等）は導入しない。

### 開発手法: TDD前提、Jest + supertest
テスト駆動開発（TDD）を前提とすることが決まっていたため、Node.js/Expressエコシステムで最も一般的で情報量の多いJestを採用。HTTPレベルの結合テストには、Expressの`app`を直接テストできサーバー起動が不要な`supertest`を採用した。

### コーディング規約: CommonJS、Prettier + ESLint、コメントは日本語
- モジュール形式はCommonJS（`require`/`module.exports`）。ESMは`package.json`への追加設定や一部ライブラリとの相性問題があるため避けた。
- Prettier（自動整形）とESLint（静的解析）を導入し、表記のぶれやバグの温床を機械的に防ぐ方針とした。ESLintはv10系のためFlat Config（`eslint.config.js`）を採用している。
- コード上のコメント（`//`や`/* */`）は日本語で書く方針とした。ただし、テストの`it`/`describe`の説明文はJestの実行結果表示に使われる文字列リテラルでありコメントではないため対象外とし、英語のままとした。

### 確認画面のデータ持ち回し: hiddenフィールド方式
確認画面から最終送信までの間、入力データをどう保持するかという設計判断。セッション（`express-session`）を使う案もあったが、依存パッケージを増やさずシンプルに実装できる「確認画面のhiddenフィールドで値を保持し、送信ボタン押下時に再送信する」方式を採用した。将来的に管理ページへ認証を追加する際は、その時点で改めてセッションの仕組みを導入する可能性がある。

### DBのstatus列: 英語コードで保持
`contacts`テーブルの`status`列は、画面表示用の日本語文字列（「新規」「対応中」「解決済み」）をそのまま格納するのではなく、英語コード（`new`/`in_progress`/`resolved`）で保持する設計とした。表示用ラベルとのマッピングは管理ページ側で1箇所にまとめる（[docs/admin.md](./admin.md)参照）。データとしての等値比較・将来的な多言語対応・コード上でのタイポ検知のしやすさを考慮した判断。

### Git運用: Conventional Commits、コミットは明示的な指示があるまで行わない
コミットメッセージはConventional Commits形式（`feat:`, `fix:`, `docs:`など）に従う方針とした。また、開発を進める中でコミットは常にユーザーからの明示的な指示（「コミットしてください」）があったタイミングでのみ行い、実装が完了しただけでは自動的にコミットしない運用とした。
